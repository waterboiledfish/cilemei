# 吃了吗 · 五人分工与迭代说明

本文档说明：（1）五人分工与协作内容；（2）第一稿与当前版本的差异及迭代原因；（3）如何接入 API 实现具体功能；（4）AI 在项目中的作用。

---

## 一、五人分工与协作

假设项目由五人协作完成，可按下表分工。实际可根据成员特长微调。

| 角色 | 负责内容 | 主要产出与协作点 |
|------|----------|------------------|
| **前端负责人（1 人）** | 页面结构、样式、交互与流式展示 | 完成 `public/index.html`、`public/style.css`、`public/main.js`；实现登录/注册表单、健康档案表单、拍照上传与图片预览的缩放/拖拽；对接后端接口（Fetch），用 ReadableStream 做 AI 结果的流式渲染；建议历史：拉取历史列表、展示并支持点击某条回填到建议区域；与后端约定接口路径与 JSON 格式。 |
| **后端负责人（1 人）** | 服务端逻辑、鉴权与数据库 | 完成 `app.js`、`server.js` 及 `api/[[...path]].js`；实现注册、登录（JWT）、`/api/me`、`/api/health`、`/api/analyze`、`/api/food-analyze`、`/api/advice-history`；设计并维护 `users`、`health_profiles`、`advice_history` 表；每次生成建议后写入建议历史，与前端约定请求/响应格式，与 AI 对接人约定 prompt 与返回结构。 |
| **AI 对接与业务逻辑（1 人）** | 调用外部 AI、设计 prompt、解析结果 | 在 `app.js` 中封装 `callAiApi`、`callAiVisionApi`；设计系统提示词与用户 prompt（健康档案、食物识别 JSON 格式）；处理 AI 返回的 JSON/非标准格式，做容错与格式化；设置超时与错误提示；与产品/前端确认「建议」的展示形式（流式/分点）。 |
| **部署与运维（1 人）** | 上线、环境配置、排错 | 配置 Vercel 项目（Framework、Build、环境变量）；配置 Supabase（建表、Connection Pooler URI）；编写/维护 `vercel.json`、`.env.example`；排查线上 500、ENOTFOUND、超时等问题；编写简单部署文档。 |
| **产品与文档（1 人）** | 需求整理、说明文档、答辩材料 | 整理功能清单与交互流程；撰写 README、FAQ、本分工与迭代说明；解释「第一稿 → 当前版」的改动原因；准备答辩用的「API 接入说明」「AI 作用说明」等表述。 |

**协作要点：**

- 前端与后端：先约定接口（URL、Method、请求体、响应体），再并行开发，用 Postman 或前端 Mock 自测。
- 后端与 AI 对接：先定好「健康建议」「食物识别」的输入/输出格式，再实现具体 HTTP 调用与错误处理。
- 部署与所有人：环境变量（如 `AI_API_KEY`、`DATABASE_URL`）由部署负责人统一在 Vercel/Supabase 配置，不写进代码；其他人提供「需要哪些变量」的清单。

## 二、第一稿与当前版本的差异及迭代原因

**第一稿与当前版本对比：**

| 方面 | 第一稿 | 当前版本 | 迭代原因简述 |
|------|--------|----------|------------------|
| 注册与数据存储 | 无完整注册流程，或数据仅存在前端（如 localStorage），刷新/换设备就丢 | 有注册/登录界面，用户与健康档案持久化存于后端（Supabase Postgres） | 需要真实「用户体系」和「历史档案」，否则无法做个性化建议和多端使用，因此补全后端鉴权与数据库。 |
| AI 输出方式 | 一次性等 AI 全部返回后再展示，耗时长、页面长时间无反馈 | 采用流式输出：后端按块写入响应，前端边收边渲染，用户能很快看到首句并持续看到新增内容 | 提升体验、降低「卡死」感；在 Vercel 等有执行时间限制的环境下，流式也能让用户尽早看到部分结果。 |
| 食物图片交互 | 仅上传/拍照，无法对图片做局部查看或放大 | 支持图片预览区的缩放（滚轮/双指）与拖拽，可放大看细节、确认识别区域 | 方便用户确认识别内容、提升对「AI 识别结果」的信任感，同时增强产品完成度。 |
| UI 整体 | 与当前版风格、布局不一致（例如更简单或不同配色） | 当前为深色主题、卡片式布局、统一组件风格 | 在补全上述功能的同时做了整体 UI 重构，统一视觉与交互规范，便于后续维护和答辩展示。 |
| 建议历史 | 无历史记录，每次生成后无法回看 | 每次文字建议与拍照识别建议自动写入 `advice_history`，可加载历史列表并点击某条重新查看 | 方便用户回顾以往建议、对比不同时期的分析结果，提升产品可用性。 |

**如何向老师/评委解释「为什么和之前做的完全不一样」：**

- **功能层面**：第一稿侧重「能不能跑通流程」，当前版本在需求上明确了三点：**（1）用户数据要落库并支持多端；（2）AI 结果要流式呈现以解决「等很久才出一大段」的问题；（3）拍照识别要支持放大缩小，方便用户查看和确认识别区域。** 为实现这些，我们补全了后端注册与数据库、改造了 AI 的调用与返回方式，并重做了前端的请求与展示逻辑，因此页面结构和交互都发生了较大变化。
- **技术层面**：第一稿可能是「纯前端 + 本地数据」或「简单后端无鉴权」的 Demo；当前版是「前端 + 鉴权后端 + 云数据库 + 外部 AI API」的完整应用。为统一技术栈、便于协作和部署，我们决定做一轮**架构与 UI 的重构**，所以看起来和初版差异大，但核心目标（健康档案 + AI 建议 + 拍照识别）一致，属于**在初版基础上的迭代与完善**，而不是推倒重做 unrelated 的项目。

## 三、我们是如何接入 API 实现具体功能的

下面按「谁提供的 API」「我们怎么调」「用来实现什么」来说明。

### 3.1 我们自己的后端 API（Node.js + Express）

- **作用**：提供用户注册、登录、健康档案的增查、以及「发起 AI 分析」的入口；所有需要用户身份的操作都经过 JWT 鉴权。
- **接入方式**：前端用 `fetch(URL, { method, headers, body })` 请求我们部署在 Vercel 上的接口（如 `https://xxx.vercel.app/api/auth/register`），后端在 `app.js` 里根据 path 和 method 路由到对应处理函数，读 `req.body` 或 `req.file`，写数据库或调外部 AI，再通过 `res.json()` 或 `res.write()` 返回结果。
- **具体对应**：
  - **注册**：`POST /api/auth/register`，body 含 `email、password、name`；后端 bcrypt 哈希密码后插入 `users` 表，返回 `token` 和 `user`。
  - **登录**：`POST /api/auth/login`，body 含 `email、password`；后端查用户、校验密码、签发 JWT，返回 `token` 和 `user`。
  - **当前用户与档案**：`GET /api/me`，Header 带 `Authorization: Bearer <token>`；后端校验 JWT 后查 `users` 和最近一条 `health_profiles`，返回 `user` 和 `profile`。
  - **保存健康档案**：`POST /api/health`，Header 带 token，body 含身高、体重、疾病史、过敏史、生活习惯；后端插入 `health_profiles`，返回最新档案。
  - **文字建议**：`POST /api/analyze`，Header 带 token，body 含健康档案字段；后端拼 prompt，调 AI 接口，拿到完整文本后写入 `advice_history`（type=text），再按块流式写回 `res`。
  - **拍照识别与建议**：`POST /api/food-analyze`，Header 带 token，body 为 `FormData`（含 `image` 和健康档案字段）；后端用 Multer 解析图片，转 base64 后与 prompt 一起发给视觉模型，解析返回的 JSON，格式化成文本后写入 `advice_history`（type=food），再按块流式写回。
  - **建议历史**：`GET /api/advice-history?limit=20&offset=0`，Header 带 token；后端按 `user_id` 查 `advice_history`，按 `created_at` 倒序返回列表（含 id、type、content_preview、content、created_at），供前端展示与点击回填。

前端在 `main.js` 里封装了带 token 的 `apiFetch` 和针对流式响应的 `fetch + response.body.getReader()`，实现「点按钮 → 调我们自己的 API → 展示结果」的完整闭环。

### 3.2 阿里云 DashScope（OpenAI 兼容）API

- **作用**：提供「根据文字生成健康建议」和「根据图片识别食物并给建议」的 AI 能力。
- **接入方式**：后端在 `app.js` 里用 `fetch` 调用 DashScope 的接口地址（环境变量 `AI_API_BASE_URL`，如 `https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`），请求体为 OpenAI 格式的 `messages`、`model`、`temperature` 等；视觉接口在 `messages` 里带 `image_url`（base64）。密钥放在环境变量 `AI_API_KEY`，不写进代码。
- **具体实现**：
  - **文字建议**：`callAiApi(prompt)` 把「用户健康档案」拼进 prompt，请求 `model: AI_MODEL`（如 qwen-vl-plus），从返回的 `choices[0].message.content` 取出文本；该文本再由 `/api/analyze` 按段落拆分，用 `res.write()` 分块写回，实现流式效果。
  - **食物识别与建议**：`callAiVisionApi({ prompt, imageBase64DataUrl })` 在 `messages` 里放一条带图片的 user message，并约定模型返回 JSON（如 `foods、nutrition_notes、advice、cautions`）；后端解析 JSON（含容错），再拼成「识别到的食物 + 营养提示 + 建议 + 特别提醒」的纯文本，由 `/api/food-analyze` 同样按块写回，前端用同一套流式逻辑展示。
- **超时与错误**：对上述 `fetch` 使用 `AbortController` 设约 9 秒超时，超时或 HTTP 非 2xx 时返回友好提示，避免长时间无响应或把底层报错直接暴露给用户。

总结：**我们先通过自己的后端 API 做鉴权和数据持久化，再在后端内部去调用 DashScope 的 API；前端只和我们自己的 API 打交道，不直接接触 DashScope，这样密钥安全、逻辑集中、也便于以后更换 AI 厂商。**

## 四、AI 在我们作业中起到了怎样的作用

可以从「功能角色」「用户体验」「技术实现」三方面说明。

### 4.1 功能角色

- **核心业务能力**：没有 AI 时，我们只能做「表单录入 + 静态展示」；有了 AI，才能根据**每个人的身高体重、疾病史、过敏史、生活习惯**生成**个性化**的饮食与生活建议，并根据**用户上传的食物照片**做识别与针对性建议。也就是说，**「个性化」和「识图」这两块核心价值，完全依赖 AI 能力。**
- **产品定位**：我们的作业定位是「健康饮食与生活方式的 AI 顾问」——AI 不是附加功能，而是区分我们项目与普通健康表单、静态科普页的关键：用户输入/拍照，AI 输出可执行建议，形成闭环。

### 4.2 用户体验

- **流式输出**：AI 生成的内容通过流式接口一段段返回并实时渲染，用户能很快看到首句、减少等待焦虑，也避免「长时间白屏再突然一大段」的体验。
- **容错与提示**：当 AI 响应过慢或超时，我们主动中断并提示「请稍后重试」；当模型返回非标准 JSON，我们做解析兜底和格式化，保证页面上始终是「可读的建议文本」而不是报错或原始 JSON。这些都是在「有 AI」的前提下，我们为体验做的设计。

### 4.3 技术实现

- **对接方式**：我们通过 HTTP 调用厂商提供的 OpenAI 兼容 API，传入 prompt 和可选图片，得到文本或 JSON，再在业务层做解析、格式化、流式输出。密钥与 URL 通过环境变量配置，便于在不同环境切换或更换模型。
- **分工体现**：AI 对接同学负责 prompt 设计、返回结构约定和容错；后端同学负责鉴权、路由和流式写回；前端同学负责调我们自己的 API 并做流式渲染。**AI 是串联「用户输入 → 后端逻辑 → 用户看到建议」的核心一环。**

**一句话概括**：在我们作业里，AI 承担了「根据用户健康档案与食物图片生成个性化、可读建议」的核心能力；我们通过自己的后端 API 接入 DashScope，并配合流式输出、超时与容错，把 AI 的输出转化为用户可直接使用的健康建议，从而完成从「数据录入 + 识图」到「建议输出」的完整产品闭环。
